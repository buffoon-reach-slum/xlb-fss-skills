# 代码检视风险检查

## 目的

在代码检视时使用本技能，提前识别项目里容易漏掉的上线风险。当前重点是接口鉴权风险：用户拥有新页面权限，但页面查询条件、下拉、页面初始化或弹窗初始化复用了其他页面接口，而该接口后端仍按原页面权限控制，导致用户进入新页面时报无权限、页面空白、按钮消失或被迫退出。

后续如果有通用的项目级 review 规则，也可以继续补充到这个技能里。

## 硬性规则：只检查本次改动

代码检视必须以真实 MR/diff 为边界，只报告本次新增或修改引入的风险。

- 先确认真实 diff 基准，再开始判断问题。
- 只检查 diff 中新增、删除、修改的文件和代码行。
- 可以阅读存量代码作为上下文，但不要把纯存量问题写成本次 MR 问题。
- 如果风险来自“改动代码调用了存量接口/存量函数”，可以报告，但必须说明触发点是本次改动。
- 如果某个文件不在本次 diff 中，不要把它列为问题文件。
- 报告问题时优先引用新增行或修改行；必要时再补充存量上下文。

## 检查 1：确认真实 Diff 基准

检视前先确认 diff 与真实 MR 源分支、目标分支一致。

- 优先使用远端 MR refs，例如 `origin/prod-gray...origin/feat-0511`。
- 不要用落后的本地目标分支做检视。
- 只报告实际出现在合入 diff 中的文件和接口。
- 如果本地目标分支与远端目标分支不同，报告中写清楚本次使用的具体 refs。

建议收集证据：

```bash
git diff --shortstat origin/target...origin/source
git diff --name-status origin/target...origin/source
git merge-base origin/target origin/source
```

## 检查 2：查询条件/下拉接口鉴权

重点检查用于搜索条件、下拉框、字典、枚举、树选择、弹窗选择器、查询控件的数据接口。

高风险接口名常见特征：

- `find`
- `page`
- `enum`
- `dict`
- `tree`
- `short`
- `available`
- `config`
- `param`

示例：

```ts
useEffect(() => {
  getSettlementCate({ settlement_cate_code: 'FEE' })
}, [])

export const getSettlementCate = (data: any) =>
  XlbFetch.post('/fss/hxl.fss.settlement.cate.page', data)
```

检视意见格式：

```md
### [P1] 查询下拉接口可能依赖其他菜单权限

- 当前页面：`src/pages/...`
- 接口：`/fss/...`
- 调用时机：搜索条件下拉加载
- 风险：用户只有当前页面权限时，可能因为该接口无权限而无法正常进入或使用页面。
- 建议：确认后端已把该接口授权到当前页面/菜单，或替换为当前页面/公共字典接口。
```

## 检查 3：页面或弹窗初始化接口鉴权

重点检查页面进入、tab 切换、抽屉打开、弹窗打开时自动调用的接口。

示例：

```tsx
useEffect(() => {
  if (!table_name) return
  void (async () => {
    const fieldsRes = await columnImportAvailableFind({ table_name, show_details: true })
    const res = await columnImportFind({ table_name, show_details: true })
  })()
}, [table_name])
```

风险：

- 用户能看到页面菜单，但页面或弹窗一打开就调用另一个受权限控制的接口。
- 用户只有新页面权限时，功能可能空白、一直 loading、无法保存，甚至页面进入失败。

检视意见格式：

```md
### [P1] 弹窗初始化接口需要确认权限归属

- 当前功能：`...`
- 接口：`/foc/...available.find`、`/foc/...import.find`
- 调用时机：弹窗打开 / 页面进入
- 风险：当前页面权限可能没有包含这些后端接口。
- 建议：确认后端权限配置已挂到当前页面。
```

## 检查 4：按钮权限名称不匹配

检查页面路由、标题、菜单属于一个功能，但 `hasAuth` 或按钮权限字符串仍引用另一个功能。

示例：

```tsx
// 页面标题：销项发票查询
{hasAuth(['供应商预付', '查询']) && <XlbButton label="查询" />}
{hasAuth(['供应商预付', '导出']) && <XlbButton label="导出" />}
```

风险：

- 用户有当前页面权限，却看不到查询、导出、保存等按钮。
- 用户有旧页面权限，反而可能看到不该操作的新页面按钮。

建议：

- 按钮权限字符串应与当前页面菜单保持一致。
- 如果产品要求复用旧权限，必须有明确的产品/后端确认。

## 检查 5：跨业务域接口复用

检查 service 方法里的接口业务域是否与当前页面业务域一致。

示例：

```ts
// 路由是销项发票查询
{
  path: '/xlb_fss/saleInvoiceCheck/index',
  title: '销项发票查询',
}

// 接口仍是供应商预付
XlbFetch.post('/fss/hxl.fss.supplieradvancepay.page', data)
```

常见不匹配信号：

- 页面/菜单是发票，但接口路径是供应商预付。
- 页面/菜单是门店，但接口路径是供应商。
- 页面/菜单是客户，但接口路径是门店费用。
- 新页面的 `server.ts` 里保留了大量复制来的旧页面动作接口。

检视意见格式：

```md
### [P0] 页面业务域与接口业务域不一致

- 路由/菜单：`销项发票查询`
- 接口业务域：`supplieradvancepay`
- 证据：
  ```ts
  XlbFetch.post('/fss/hxl.fss.supplieradvancepay.page', data)
  ```
- 风险：数据来源和后端权限域都可能错误。
- 建议：改为当前功能接口；如果确实复用旧接口，需要同步调整页面命名、菜单和权限说明。
```

## 检查 6：XlbFetch 错误处理行为

在 FSS 项目中，`XlbFetch` 已经处理常见请求异常和业务报错。检视时重点关注接口权限归属和页面可用性，不要重复要求泛用错误提示。

需要检查：

- 普通 `XlbFetch` 调用后又手动 `throw new Error(res.msg)`。
- 普通非 0 响应又重复 `message.error(res.msg)`。
- 新增自定义请求客户端绕开 `XlbFetch`。
- 可选查询接口失败后，被写成阻断页面进入的流程。

检视建议：

- 不要要求重复加泛用错误提示。
- 优先确认接口是否授权到当前页面权限下。
- 如果确实需要手动处理，只针对具体业务场景处理。

## 输出规则

每条问题尽量包含：

- 当前页面或功能
- 本次改动触发点：新增文件、修改行、路由新增、查询条件新增、弹窗新增等
- 接口路径
- 调用时机：查询下拉、页面进入、弹窗打开、按钮操作、导出
- 简短代码证据
- 对“只有当前页面权限的用户”的影响
- 修复建议或后端权限确认点

如果无法从代码直接证明后端权限配置，不要写成确定 bug，写成“需要后端权限确认”。

不要输出不在本次 diff 中的历史问题，除非它被本次改动重新暴露或接入到新的页面/流程中。
